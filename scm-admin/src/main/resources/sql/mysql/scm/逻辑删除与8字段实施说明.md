# 逻辑删除与 8 字段（create_by/create_time/update_by/update_time/del_flag/del_time/del_by/tenant_id）实施说明

## 一、已完成

### 1. column.sql
- 已在 `column.sql` 中为下列表增加 **CALL add_table_column** 语句，补充缺失的：`del_flag`、`del_time`、`del_by`、`tenant_id`（及部分表的 `create_by`/`update_by`）。
- 涉及表：scm_certificate_config, scm_certificate_type, scm_delivery, scm_delivery_detail, scm_delivery_invoice, scm_hospital, scm_hospital_supplier, scm_hospital_user, scm_manufacturer, scm_material_category, scm_material_dict, scm_order, scm_order_detail, scm_product_certificate, scm_purchase_statistics, scm_settlement, scm_settlement_detail, scm_supplier, scm_supplier_certificate, scm_supplier_code_mapping, scm_supplier_user, sys_dept, scm_tenant, sys_user（含 user_source）。
- 执行顺序：先执行 `table.sql`，再执行 `column.sql`（依赖存储过程 `add_table_column`）。

### 2. BaseEntity（scm-common）
- 已增加字段：`delFlag`、`delTime`、`delBy`、`tenantId`，及对应 getter/setter。
- 所有继承 `BaseEntity` 的实体自动具备：createBy, createTime, updateBy, updateTime, remark + 上述 4 个字段。
- 已从 **Hospital、Supplier、MaterialDict、MaterialCategory** 中移除重复的 `delFlag` 定义，统一使用 BaseEntity。

### 3. 示例模块：CertificateConfig（证件配置）
- **Mapper XML**  
  - resultMap / selectVo 已包含：del_flag, del_time, del_by, tenant_id。  
  - 列表/按类型查询：WHERE 增加 `(del_flag = '0' OR del_flag IS NULL)`。  
  - insert：写入 del_flag（默认 '0'）、tenant_id。  
  - update：可更新 del_flag, del_time, del_by, tenant_id。  
  - 原 delete 已改为 **逻辑删除**：`update ... set del_flag='2', del_time=sysdate(), del_by=#{delBy}`。
- **Mapper 接口**  
  - `deleteCertificateConfigByConfigId(configId, delBy)`、`deleteCertificateConfigByConfigIds(configIds, delBy)` 增加参数 `delBy`。
- **Service**  
  - insert 时若 `delFlag==null` 则设为 `"0"`。  
  - 删除方法调用 Mapper 时传入 `ShiroUtils.getLoginName()` 作为 `delBy`。

## 二、其余表需按同一模式补齐

对下列每个模块，按 **CertificateConfig** 的方式做三处修改：

### 1. Mapper XML
- 在 **resultMap** 和 **selectXxxVo** 中增加：`del_flag`、`del_time`、`del_by`、`tenant_id`（若表结构中已有 create_by 等，则只补这 4 个）。
- 所有 **列表/条件查询**：WHERE 中增加 `AND (del_flag = '0' OR del_flag IS NULL)`（若表有 del_flag）。
- **insert**：列与值中增加 `del_flag`（默认 '0'）、`del_time`、`del_by`、`tenant_id`（按需）。
- **update**：`<set>` 中增加对 `del_flag`、`del_time`、`del_by`、`tenant_id` 的更新（按需）。
- 将原来的 **delete** 改为 **update**：  
  `update 表 set del_flag='2', del_time=sysdate(), del_by=#{delBy} where 主键条件`。  
  批量删除同理。

### 2. Mapper 接口
- 所有“删除”方法增加参数：`@Param("delBy") String delBy`，在 Service 中传入当前操作人（如 `ShiroUtils.getLoginName()`）。

### 3. Service
- **insert**：  
  - 若实体继承 BaseEntity，一般已有 createBy/createTime（由 Controller 或 AOP 填充）；  
  - 若未设置 `delFlag`，则 `entity.setDelFlag("0")`。
- **update**：  
  - 由 Controller 设置 updateBy/updateTime，或在本层设置。
- **删除**：  
  - 不再调用物理 delete，改为调用上述“逻辑删除”的 update 方法，并传入 `delBy`。

### 4. 前端 / Controller 填充策略
- **createBy / createTime**：在 **Controller 的“新增”方法** 中，调用 Service 前执行：  
  `entity.setCreateBy(getLoginName());`、`entity.setCreateTime(new Date());`（或使用项目统一工具）。
- **updateBy / updateTime**：在 **Controller 的“修改”方法** 中，调用 Service 前执行：  
  `entity.setUpdateBy(getLoginName());`、`entity.setUpdateTime(new Date());`。
- 若使用 AOP 或拦截器统一设置创建人/修改人，则保持现有方式，确保 insert/update 时这 4 个字段有值即可。

## 四、执行与验证建议

1. 数据库：在目标库先执行 `column.sql`，确认无“列已存在”类错误（add_table_column 会跳过已存在列）。
2. 编译：执行 `mvn compile`，确认实体与 Mapper 接口无报错。
3. 功能：对已改造的 **CertificateConfig** 做一次“新增 / 修改 / 删除”，检查：  
   - 列表中不再出现已删数据；  
   - 删除后表中对应行为 `del_flag='2'`，且 `del_time`、`del_by` 有值。

其余模块按上表逐个改造即可复用同一套策略。
